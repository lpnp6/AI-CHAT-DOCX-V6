# 基于 LLM 的 DOCX 自动化填写方案

## 1. 调研背景

在实际业务中，大量文档仍以 Word 模板的形式流转，例如简历、申报材料、财务声明、访视报告等。这类文档通常已经具备固定版式，但填写过程仍高度依赖人工复制、粘贴和校对，效率低且容易出错。

基于上述问题，本次调研尝试验证一种新路线：**直接读取 DOCX 内部 XML，由大语言模型理解文档结构并生成 XPath 指令，再由本地执行器完成填写。**

## 2. 调研目标

本次调研主要希望回答以下问题：

- 大模型是否能够从 DOCX 的 XML 部件中理解文档结构
- 大模型是否能够根据自然语言填写要求生成可执行的 XPath 文本替换指令
- 这种方案在多轮修复场景下是否能达到最优
- 当前方案是否只适合文本填写，还是可以进一步扩展到样式修改

## 3. 技术路线设计

DOCX 文件本质上是一个压缩包。解压后可以看到大量 XML 文件，其中正文内容主要位于：

- `word/document.xml`

该方案采用的是一个基于 DOCX XML 的 Agent Loop：

1. `package` ：解包 DOCX，提取 XML 部件
2. `llm` 接收用户需求、解包后的 XML 内容，以及当前执行上下文
3. `llm` 返回一组 `instruction` 指令
4. `executor` 在本地执行这些指令
5. `feedback` 汇总执行结果、失败项和锁定成功项
6. 如果仍有失败，则将反馈继续送回 `llm`，进入下一轮修复
7. 如果没有失败，则结束循环并重新打包输出 DOCX

![Agent Loop 流程图](./截屏2026-03-16%2014.28.58.png)

该方案的核心特点是：

- 不依赖模板预埋占位符
- 不要求用户先提供结构化字段映射
- 将文档结构理解交给 LLM
- 将实际写入保留在本地可控执行器中
- 通过反馈驱动多轮修复，而不是只依赖一次生成，**填写工作是一个逐渐收敛的过程**

其中，大模型上下文主要包括：

- 当前 XML 部件内容
- 上一轮失败指令
- 已成功写入并锁定的 XML 节点
- 用户需求

LLM 返回的核心指令格式如下：

```json
{
  "instructions": [
    {
      "type": "set_text",
      "xpath": "./w:body/w:p[1]/w:r[1]/w:t[1]",
      "text": "replacement text"
    }
  ]
}
```

当前只支持一种指令类型：

- `set_text`

## 4. 验证能力

### 4.1 文本填写能力

当前原型已经能够完成基于自然语言的文本填写。例如：

- 将模板中的姓名、学历、职位、单位等信息填入指定位置
- 在表格结构中定位对应单元格
- 在段落和文本运行层级完成替换

这说明 LLM 在“从 XML 中找出可写位置并返回 XPath”这一任务上具备初步可用性。

### 4.2 多轮失败修复能力

对于首轮失败的情况，系统支持将失败项反馈给模型，要求模型重新生成指令。这种方式在实际测试中能够修复部分 XPath 错误，成功为大模型不确定性做了一定的兜底。

## 5. 结论

### 5.1 该方案在文本填写上是可行的

LLM 能够基于 `document.xml` 理解基本结构，并输出可执行的 XPath 文本替换指令。对表格类短模板文档尤其有效。

### 5.2 当前方案对模板文件的依赖较大

大模型需要完整并能成功解包的 DOCX 文档，然后基于正文部分生成严格规范的修改指令。不能强行生成 DOCX 文档，如果需要从 0 开始生成文档，可以从底层注入默认的模板文件。

### 5.3 多轮修复是必要能力

仅靠“首轮生成一次指令”很难覆盖真实文档场景。多轮修复能有效解决大模型幻觉，为大模型本身的不确定性兜底。

### 5.4 工具执行约束和 prompt 约束同等重要

从 prompt 到工具执行必须对齐。例如，只在提示词中告诉模型“不要覆盖成功内容”并不够。还要在执行层显式拒绝覆盖成功内容。还要将已经犯错并输出了覆盖成功内容的指令反馈给大模型。

## 6. 字体样式修改的可行性分析

是否能进一步支持字体样式识别以及修改，结论如下：

### 6.1 大模型能否从 XML 中读出样式

可以读出一部分。

在实际 `document.xml` 中可以直接看到很多样式节点，例如：

- `w:rPr`
- `w:rFonts`
- `w:sz`
- `w:b`
- `w:i`
- `w:u`
- `w:color`

因此，如果样式是直接写在文本运行节点上的，模型可以读出。

### 6.2 样式问题的复杂性高于文本问题

因为 Word 的样式不只存在于 `document.xml`，还可能来自：

- `styles.xml`
- `theme/theme1.xml`
- `numbering.xml`
- 页眉页脚等其他内容部件

因此，“读到局部样式”不等于“理解最终生效样式”。


## 7. 主要局限

当前方案仍存在明显边界：

直接将 XML 语言交给大模型，是非常信任大模型能力的行为。当大模型能力相对不足时，可能会出现 Agent Loop 进入较长循环、导致上下文超出窗口大小的情况，也可能出现 Token 浪费。所以对于结构非常复杂、文本量极大的 DOCX 文档，目前只有 gpt-5.4 能凭借其非常大的上下文窗口以及可信的推理能力，才勉强胜任。


## 8. 优点

综合调研结果，可以得到以下结论：

### 8.1 框架简洁，可扩展性强

整个框架较为简单，实现复杂功能时，可以针对指定模块进行优化。例如挂载复杂的修改工具，优化特定领域 prompt，不会出现牵一发动全身的问题。可以将其封装为 Skill，也可以搭建 MCP 服务。

### 8.2 大模型对任务的掌控更大，也能受到严格约束

大模型能从底层读懂 DOCX 文档，生成自由度高的工具调用指令。同时有执行层的强约束，不会让大模型生成的无效指令或错误修改实际应用于文档。

## 9. 后续建议

假如方案有后续的话，可以从以下方面迭代：

### 引进 XML 的中间层表示

可以让大模型获取更加结构化的文档信息，使得大模型的不确定性进一步降低，还能同时降低上下文大小。

### 构建 docx 的部件关系图

DOCX 由 XML 部件和二进制部件组成，同时还保留了逻辑结构的表示。所以可以构建部件关系图，这是可行并已经被验证了。构建部件关系图理论上能实现对 DOCX 文档的任意修改，包括图片更换、模板修改。

### 在 Agent 上游插入任务分流 Router

面对各种各样的文档，直接解包并交给大模型是不可行的，所以 Router 是一个非常重要的部件。它的首要任务是解包文档后结合用户需求判断任务难度，然后根据难度将任务分流到不同工作流。例如，简历模板这类短模板文档填写，以上方式能高效完成；而对于长模板文档，或涉及文档结构、字体样式等更加底层的高难度任务，就需要分流到更复杂的工作流，例如长文档分块、查询部件关系图、精准定位到需要修改的 XML 部件。同时，WPS 和 Word 会出现不兼容的情况，主要是 WPS 支持的一些 DOCX 文档存在私有部件或者节点容器，所以也需要 Router 对其进行区别，这样能对下游的 Agent Prompt、Workflow 进行挑选。
